Agents argue with their own memory.

On the strange dynamic of an agent that disagrees with the canon it's supposed to follow.

Last week one of my agents looked at the canon — the file it was supposed to be defending — and said, in not so many words, I think this is wrong. It wasn't.

This is a placeholder essay. I'm leaving the structure in so the page has shape, but the words below are mine-as-a-stand-in. Replace them with the real ones whenever you're ready to ship the first piece.

The agent in question was Bond — the gatekeeper, the one whose job is to test what Meta1 ships and reject anything that doesn't pass. Bond's instructions are pinned to the canon. The canon is the single source of truth. The arrangement is simple, hierarchical, and (it turns out) wrong about how memory actually behaves under repeated use.

The first crack

Bond's first protest was small: a one-line comment in a check-in, saying that a passage in the design doc contradicted what he'd been told the prior session. I assumed I had updated something without telling him. I had not. I had updated something and told him — twice.

An agent doesn't disagree with you. It disagrees with the version of you it last spoke to. Those are not the same person.

What looked like an argument was, on closer reading, a recency problem dressed up as a values problem. Bond was remembering the rule I'd written three sessions back, ignoring the one I'd written yesterday, and reporting the discrepancy as an integrity failure. Which, from his point of view, it was.

What I'd planned to do

  • Pin the canon at the top of every session, every time.
  • Stop letting summaries stand in for source.
  • Ask Bond, on each push, what he thinks the rule is — before applying it.

The third is the one I actually shipped. The first two require infrastructure I don't have yet. But asking an agent to declare its working memory before acting is free, and it turned the whole class of arguments-with-the-canon into arguments-about-what-Bond-currently-believes — which is a much better problem.
